# 功能規格：CyberBrick 上傳模式設定

**功能分支**：`[059-cyberbrick-upload-modes]`
**建立日期**：2026-05-27
**狀態**：草稿
**使用者輸入**：「為 CyberBrick 提供明確的上傳模式設定入口，讓使用者可在 USB 與 OTA 之間切換；OTA 需透過專用設定完成裝置配對、Wi‑Fi 資訊與目標裝置選擇，避免每次上傳都跳出詢問，並以對小朋友友善的方式提供附近 Wi‑Fi 名稱建議與手動輸入能力。」

## Clarifications

### Session 2026-05-28

- Q: 第一次 OTA 配對的入口要怎麼定義？ → A: 首次 OTA 配對只能透過 USB 完成；完成後才可切到 OTA 日常上傳。
- Q: 多台已配對 CyberBrick 的主要辨識鍵要怎麼定義？ → A: 以 `deviceId` 為唯一主鍵，`friendlyName` 只作為顯示名稱，可重複。
- Q: 已配對裝置清單與 OTA 設定應該保存在哪個範圍？ → A: 非敏感 OTA 設定與已配對裝置清單以專案為範圍保存；Wi‑Fi 密碼、token 等敏感資料只保存在使用者本機的安全儲存中。
- Q: 當目前是 `OTA` 模式，但目標裝置無法連線時，系統應如何處理？ → A: 不自動 fallback；顯示明確錯誤與下一步，讓使用者自己選擇重新設定、改選裝置或切回 USB。
- Q: OTA 設定畫面中的 Wi‑Fi 名稱建議，應該從哪裡取得？ → A: 由 CyberBrick 裝置本身掃描並回傳 Wi‑Fi 名稱建議。

## 使用者情境與測試（必填）

### 使用者故事 1 - 先選好上傳模式，不要每次都被打擾（優先級：P1）

作為使用 CyberBrick 的學生或老師，我希望在編輯畫面裡有一個固定的上傳設定入口，可以先決定這個專案目前要用 `USB` 還是 `OTA` 上傳，這樣我按下上傳時就會直接照目前模式執行，不會每次都被重複詢問要不要改用 OTA。

**優先原因**：這直接決定日常使用體驗是否順暢，尤其是面向小朋友時，避免重複打斷比新增更多自動判斷更重要。

**獨立測試**：只實作此故事時，使用者仍應能在不設定 OTA 的情況下正常完成 USB 上傳；若改成 OTA 模式，後續連續上傳應直接依目前模式執行，而不是每次再詢問一次。

**驗收情境**：

1. **假設** 使用者第一次在 CyberBrick 專案中開啟上傳設定，**當** 尚未做任何變更，**則** 系統應顯示 `USB` 為預設上傳模式。
2. **假設** 使用者已把上傳模式改成 `OTA` 並完成儲存，**當** 使用者再次按下上傳，**則** 系統應直接以 OTA 模式執行，而不是再次詢問是否改用 OTA。
3. **假設** 使用者已完成 OTA 設定但目前選擇 `USB` 模式，**當** 使用者按下上傳，**則** 系統應沿用既有 USB 上傳流程，且不得要求填寫 OTA 專用資料。
4. **假設** 使用者把模式從 `OTA` 切回 `USB`，**當** 使用者進行下一次上傳，**則** 系統應只使用 USB 流程，不得偷偷保留 OTA 作為隱藏預設。

---

### 使用者故事 2 - 第一次設定 OTA 時有清楚引導與 Wi‑Fi 協助（優先級：P2）

作為第一次想使用無線上傳的人，我希望系統在我選擇 OTA 模式時，能清楚引導我先透過 USB 完成首次可信任配對，再完成裝置命名與 Wi‑Fi 設定，並由裝置本身掃描列出它目前看得到的 Wi‑Fi 名稱給我選，必要時也能手動輸入，這樣我不需要把 Wi‑Fi 連線邏輯塞進學生作品，卻仍能把裝置準備好進行 OTA 上傳。

**優先原因**：若第一次設定過程模糊或繁瑣，使用者很容易放棄 OTA；把網路與配對設定從作品邏輯中分離，才能讓 OTA 成為真正可用的上傳方式。

**獨立測試**：只實作此故事時，使用者應能在一個專用設定流程中完成首次 OTA 準備，包含裝置識別、Wi‑Fi 設定與基本完成條件確認，而不必修改學生作品內容。

**驗收情境**：

1. **假設** 使用者選擇 `OTA` 模式但尚未完成首次設定，**當** 使用者儲存或準備上傳，**則** 系統應引導使用者先透過 USB 連線完成首次 OTA 配對與設定，而不是直接失敗。
2. **假設** CyberBrick 裝置本身可提供附近的 Wi‑Fi 名稱建議，**當** 使用者開啟 OTA 網路設定，**則** 系統應顯示由該裝置掃描得到的可選擇 Wi‑Fi 名稱清單，並允許使用者重新整理建議。
3. **假設** 想使用的 Wi‑Fi 沒有出現在建議中，**當** 使用者手動輸入 Wi‑Fi 名稱與密碼，**則** 系統仍應允許完成 OTA 設定。
4. **假設** 使用者已完成 OTA 設定，**當** 使用者回到上傳設定畫面，**則** 系統應保留先前輸入的專案層級非敏感設定與已配對狀態，避免每次重新開始。

### 使用者故事 3 - 同一個教室有多台裝置時，仍能選對目標（優先級：P3）

作為在同一個 Wi‑Fi 環境裡管理多台 CyberBrick 的老師或學生，我希望系統能記住多台已配對裝置，並以穩定的 `deviceId` 搭配 `friendlyName` 提供足夠清楚的識別資訊，讓我知道目前會上傳到哪一台，這樣我就不會把程式傳到隔壁同學的裝置上。

**優先原因**：多裝置辨識是 OTA 真正落地的關鍵；若使用者無法有信心地選中正確裝置，OTA 會比 USB 更讓人緊張。

**獨立測試**：只實作此故事時，使用者應能在同一個專案中看到多台已配對裝置、指定主要目標，並在名稱重複或裝置暫時離線時仍知道下一步。

**驗收情境**：

1. **假設** 同一個專案已有兩台以上已配對 CyberBrick，**當** 使用者開啟上傳設定，**則** 系統應顯示可辨識的裝置清單，並允許指定主要上傳目標。
2. **假設** 兩台裝置使用相同或相似名稱，**當** 使用者選擇上傳目標，**則** 系統應仍能以 `deviceId` 等唯一識別資訊清楚區分它們。
3. **假設** 使用者已指定主要 OTA 目標，**當** 使用者按下上傳，**則** 系統應把程式傳送到該目標，而不是改傳到最近被發現的其他裝置。
4. **假設** 目前選定的裝置暫時無法連線，**當** 使用者嘗試 OTA 上傳，**則** 系統應清楚指出是哪一台裝置無法連線，並提供重新選擇、重新檢查或改用 USB 的下一步建議，但不得自動改走 USB 上傳。

### 邊界情境

- 當使用者切換到 `OTA` 模式，但尚未透過 USB 完成首次配對或網路設定時，系統必須先阻止上傳並引導完成必要設定。
- 當 CyberBrick 裝置本身掃描得到的附近 Wi‑Fi 名稱建議為空、掃描結果不完整，或教室使用隱藏網路時，系統必須仍允許手動輸入網路名稱。
- 當同一個專案中有多台裝置且 `friendlyName` 重複時，系統必須以 `deviceId` 等名稱以外的資訊幫助使用者分辨正確目標。
- 當使用者已完成 OTA 設定，但裝置換了 IP、暫時離線或移到不同網路時，系統必須在上傳前給出明確的修正引導，而不是靜默失敗或自動切回 USB。
- 當使用者從 `OTA` 切回 `USB` 後，系統必須完全回到原本的有線上傳體驗，不得持續跳出 OTA 相關提示。
- 當使用者從未打算使用 OTA 時，系統必須不增加現有 USB 流程中的步驟負擔。

## 需求（必填）

### 功能需求

- **FR-001**: 系統必須在 CyberBrick 編輯體驗中提供固定可見的上傳設定入口，讓使用者可在按下上傳前先檢查或變更上傳模式。
- **FR-002**: 系統必須提供至少兩種明確的上傳模式：`USB` 與 `OTA`。
- **FR-003**: 對新的 CyberBrick 專案，系統必須預設使用 `USB` 上傳模式。
- **FR-004**: 即使使用者已完成 OTA 配對，系統也不得自動把上傳模式改成 `OTA`；只有使用者明確切換後才可改變目前模式。
- **FR-005**: 系統必須以專案為範圍記住目前選定的上傳模式、已配對裝置清單與其他非敏感設定，讓使用者重新開啟同一專案後不需重新設定基本資料。
- **FR-006**: 在 `USB` 模式下，系統必須維持既有的有線上傳體驗，且不得要求填寫 OTA 專用資料。
- **FR-007**: 當使用者選擇 `OTA` 模式且尚未完成首次設定時，系統必須提供清楚的引導流程，要求使用者先透過 USB 完成首次可信任配對與基本準備。
- **FR-008**: OTA 設定流程必須為每台裝置建立穩定的唯一識別 `deviceId`，並允許使用者指定僅供顯示的 `friendlyName`，以避免同一網路中多台裝置時混淆目標。
- **FR-009**: OTA 設定流程必須讓使用者查看由 CyberBrick 裝置本身掃描並回傳的 Wi‑Fi 名稱建議，並允許使用者主動重新整理建議清單。
- **FR-010**: 如果想連接的 Wi‑Fi 沒有出現在建議清單中，系統必須允許使用者手動輸入網路名稱。
- **FR-011**: 系統必須讓使用者輸入 Wi‑Fi 密碼與配對憑證等敏感資料，但不得把這些資料寫進學生作品內容或專案可分享設定中。
- **FR-012**: 系統必須只將 Wi‑Fi 密碼、token 與其他敏感憑證保存在使用者本機的安全儲存中，而不得跟隨專案同步或匯出。
- **FR-013**: 完成 OTA 設定後，系統必須仍維持原本的 `USB` 模式，直到使用者自己明確切換成 `OTA`。
- **FR-014**: 系統必須允許同一個專案記住多台已配對 CyberBrick，並以 `deviceId` 作為每台裝置的真正主鍵，讓使用者指定目前主要的 OTA 上傳目標。
- **FR-015**: 當多台裝置的 `friendlyName` 相同或相似時，系統必須仍能提供清楚的 `deviceId` 或等效唯一識別資訊，避免使用者選錯目標。
- **FR-016**: 系統必須提供足以辨識目標裝置的最近狀態資訊，讓使用者能判斷目前選到的是哪一台裝置。
- **FR-017**: 在 `OTA` 模式下，使用者按下上傳後，系統必須直接依目前設定的目標開始 OTA 上傳，不得每次再次詢問是否改用 OTA，也不得在失敗時自動改走 USB。
- **FR-018**: 若 `OTA` 模式所需資料不完整、裝置未完成配對或目前無法連線，系統必須在開始上傳前清楚指出缺少或失效的資訊，並引導使用者修正。
- **FR-019**: 系統必須避免持續背景刷新 Wi‑Fi 名稱建議，以免造成畫面干擾；建議更新應發生在使用者開啟設定或主動要求重新整理時。
- **FR-020**: 使用 OTA 上傳時，系統必須不要求學生為了上傳而在作品中加入 Wi‑Fi 連線積木或其他非教學必要邏輯。
- **FR-021**: 未啟用 OTA 的使用者必須能完全沿用原本上傳流程，不因新功能增加額外阻礙。
- **FR-022**: 當 OTA 上傳失敗時，系統必須提供可理解的失敗原因與下一步建議，例如重新選擇裝置、檢查網路，或暫時改用 USB；是否切換回 USB 必須由使用者明確決定。

### 主要資料實體（若功能涉及資料則必填）

- **上傳模式設定**：代表某個 CyberBrick 專案目前採用的上傳方式、是否已完成 OTA 準備，以及以專案為範圍保存的非敏感上傳偏好設定。
- **已配對裝置**：代表一台已與專案建立信任關係的 CyberBrick，包含作為真正主鍵的 `deviceId`、可重複的顯示名稱 `friendlyName`、最近狀態與是否為主要上傳目標；其非敏感描述跟隨專案保存。
- **Wi‑Fi 設定項目**：代表 OTA 所需的網路資訊，包含由 CyberBrick 裝置本身掃描並回傳的 Wi‑Fi 名稱建議、手動輸入的網路名稱，以及只保存在本機安全儲存中的敏感憑證。
- **OTA 準備狀態**：代表目前專案是否已完成 OTA 所需的必要步驟，以及若未完成時還缺少哪些條件。

## 成功標準（必填）

### 可量測成果

- **SC-001**: 90% 以上第一次使用 OTA 的目標使用者，能在 5 分鐘內完成首次 OTA 設定並成功送出至少一次無線上傳。
- **SC-002**: 在 `USB` 模式下，100% 的上傳流程都不會跳出 OTA 切換詢問，也不需要填寫任何 OTA 專用欄位。
- **SC-003**: 在同一個專案至少有 2 台已配對裝置的驗證情境中，95% 以上的使用者能在第一次選擇時選中正確上傳目標。
- **SC-004**: 在兒童導向的連續上傳情境中，90% 以上已完成模式設定的使用者，不會在第二次之後的連續上傳中再次遇到模式選擇對話。
- **SC-005**: 當 Wi‑Fi 名稱建議清單為空或不完整時，90% 以上的使用者仍能透過手動輸入完成 OTA 網路設定，而不需要額外求助。
- **SC-006**: 在 OTA 模式設定不完整或目標不可用的情境中，95% 以上的上傳嘗試都能在真正開始傳送前收到明確且可行動的修正指引。

## 假設

- OTA 的首次準備會在裝置可被安全設定的情境下完成，之後才進入日常的無線上傳使用。
- 使用者在第一次啟用 OTA 時，手邊可暫時使用 USB 連線完成首次配對。
- 專案層級記住上傳模式與已配對裝置的非敏感資訊，符合教室與作品分開管理的使用情境。
- 敏感憑證只保存在本機安全儲存中，即使專案被複製或分享也不會自動帶出。
- 第一版只聚焦明確的 `USB` 與 `OTA` 兩種模式；自動判斷何時切換的智慧模式不在此版本範圍內。
- Wi‑Fi 名稱建議以 CyberBrick 裝置本身的掃描結果為準，而不是使用者電腦目前可見的網路列表。
- Wi‑Fi 名稱建議有可能偶爾不完整，因此手動輸入屬於基礎能力而非例外流程。
- 現有的 USB 上傳仍會持續可用，並作為未完成 OTA 設定或暫時無法無線連線時的安全退路。
