# Feature Specification: PlatformIO Guided Repair

**Feature Branch**: `[058-platformio-guided-repair]`  
**Created**: 2026-05-25  
**Status**: Draft  
**Input**: User description: "強化既有 PlatformIO 狀態檢查，讓同一個介面除了偵測外，還能提供以單一主按鈕啟動的自動修復流程、在安全範圍內自動嘗試修復，並把每次嘗試的成功與失敗記錄下來，供後續 AI 修復判斷與 issue 草稿整理使用。"

## Clarifications

### Session 2026-05-25

- Q: `自動修復` 按鈕在 v1 允許動到哪個層級？ → A: 允許在使用者空間內執行有明確說明且可審計的修復動作，但不碰系統層。
- Q: `自動修復` 一次按下去要跑到多遠？ → A: 一次點擊可執行同一條主要修復流程中的有限步驟序列，例如最多 2–3 個安全步驟；成功或遇到阻塞失敗就停止。
- Q: `自動修復` 在執行前需要什麼程度的使用者確認？ → A: 每次執行前先做一次輕量確認，列出這次要跑的步驟序列與影響範圍，使用者確認後才開始；不要求每個步驟逐一確認。
- Q: AI 修復封包與 issue 草稿的隱私預設應該是什麼？ → A: 預設自動遮罩已知敏感資訊（如使用者名稱、絕對路徑、代理/帳號資訊、疑似 token），必要時才由使用者明確選擇揭露更多細節。
- Q: 修復歷程要保存到什麼範圍？ → A: 修復歷程以工作區為範圍持久保存，可跨面板重開與 VS Code 重啟保留，直到使用者清除或環境指紋明顯改變。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在同一個 PlatformIO 狀態檢查裡啟動自動修復並追蹤結果 (Priority: P1)

作為被環境問題卡住的使用者，我希望在既有的 PlatformIO 狀態檢查介面中，不只看到工具鏈有沒有問題，還能透過一顆清楚的主按鈕啟動自動修復，讓系統先在安全範圍內自動嘗試同一條主要修復流程中的有限步驟序列，同時保留每次嘗試的成功與失敗紀錄，這樣我就不用自己猜下一步要做什麼，也不必在失敗後重新回想前面試過哪些路徑。

**Why this priority**: 這是整個功能的核心價值。只要使用者能在同一個地方看到診斷結果，並用單一主按鈕啟動系統自動嘗試最值得先做的一小段修復流程，就能明顯降低卡關時間與求助成本。

**Independent Test**: 在工具鏈處於 degraded 或 unavailable 的情境下開啟既有 PlatformIO 狀態檢查；若只實作此故事，使用者仍應能看懂主要問題類型、點擊單一主按鈕啟動至少一部分安全自動修復、看到每次嘗試的結果紀錄、知道失敗後再做什麼，以及修完後如何重新驗證。

**Acceptance Scenarios**:

1. **Given** 使用者開啟 PlatformIO 狀態檢查且結果不是完全正常, **When** 診斷完成, **Then** 介面應在同一個檢查體驗中顯示主要問題分類、推薦修復流程與可啟動的主 `自動修復` 按鈕，而不是只顯示狀態清單。
2. **Given** 診斷同時出現多個異常訊號, **When** 系統選出主要問題類型, **Then** 介面應優先突出一條推薦修復路徑，同時保留底層證據供使用者查閱。
3. **Given** 某一條推薦修復路徑包含可在介面內完成的安全步驟, **When** 使用者點擊主 `自動修復` 按鈕, **Then** 系統應先顯示一次輕量確認，清楚說明將執行的有限步驟序列與影響範圍，並在使用者確認後才開始執行。
4. **Given** 使用者在輕量確認畫面中選擇不執行, **When** 這次修復尚未開始, **Then** 系統應保留目前診斷與推薦資訊，但不得實際變更環境。
5. **Given** 同一條主要修復流程中的步驟序列正在執行, **When** 其中一個步驟已使問題達到成功條件或遇到阻塞失敗, **Then** 系統應停止後續自動嘗試並清楚說明停止原因。
6. **Given** 使用者已執行一個或多個修復步驟, **When** 某次步驟完成、失敗或中止, **Then** 系統應保留可讀的嘗試紀錄，包含試過哪些方法、各自結果以及是否仍值得再次嘗試。
7. **Given** 使用者已嘗試修復, **When** 使用者重新執行檢查, **Then** 介面應清楚反映問題是否改善、未變或惡化。
8. **Given** 使用者關閉並重新開啟面板，或重新啟動 VS Code 回到同一工作區, **When** 工作區環境未明顯改變且使用者未手動清除歷程, **Then** 系統應仍能顯示先前的修復歷程，避免重複試錯。

### User Story 2 - 複製 AI-ready 修復封包以取得高品質協助 (Priority: P2)

作為需要向 AI、教師或維護者求助的使用者，我希望複製出去的摘要不是只有錯誤訊息，而是一份包含事實證據、已嘗試方法與回答格式要求的修復封包，這樣對方回覆時能直接說明這次採用的修法與理由，也能避開前面已失敗的路徑。

**Why this priority**: 單純的診斷文字常常會讓協助者回覆泛化建議。把證據與回答契約一起帶出去，才能把一次性求助轉成可重用的修復知識。

**Independent Test**: 在有警告或錯誤的情境中複製摘要；若只實作此故事，使用者應能把內容貼給 AI 或協助者，並在第一次回應中取得結構化的根因判斷、修法、驗證方式與可重用 heuristic。

**Acceptance Scenarios**:

1. **Given** 診斷結果已產生, **When** 使用者複製摘要, **Then** 複製內容應同時包含證據區塊、已嘗試修復歷程與 AI 任務區塊，而不是只列出工具狀態。
2. **Given** 診斷結果中存在警告或錯誤, **When** AI 根據摘要回應, **Then** 摘要應促使 AI 區分已觀察事實與推測，並回報最可能根因、實際採用修法、驗證方法、可重用 heuristic，以及接下來仍可嘗試的路徑。
3. **Given** 使用者複製 AI-ready 修復封包, **When** 摘要中包含使用者名稱、絕對路徑、代理/帳號資訊或疑似敏感字串, **Then** 系統應預設先遮罩這些資訊，並只在使用者明確選擇時才揭露更多細節。
4. **Given** 不同語系的使用者都可能複製摘要, **When** 摘要被貼到 AI 或維護流程中, **Then** 內容應保有穩定欄位結構，讓後續整理與比較不依賴單一語系文案。

### User Story 3 - 經人類核准後把可重用修復經驗整理成 issue 草稿 (Priority: P3)

作為維護者或進階使用者，我希望在 AI 協助修復後，只有那些值得產品化沉澱的案例才會在人類核准後整理成 issue 草稿，這樣專案能保留高訊號經驗，而不會被一次性環境雜訊淹沒。

**Why this priority**: 開源專案需要知識沉澱，但也需要控制 backlog 噪音。把「是否值得進 backlog」交給人類最後把關，能兼顧效率與治理品質。

**Independent Test**: 在修復成功且人類認可案例具有可重用價值時，建立 issue 草稿；若只實作此故事，使用者仍應能完成案例整理、人工確認、重複/隱私檢查與 issue 草稿輸出，而不必依賴自動公開開單。

**Acceptance Scenarios**:

1. **Given** 一個案例已完成診斷與修復，且人類判定該案例值得沉澱, **When** 使用者選擇建立後續追蹤, **Then** 系統應產生包含問題摘要、已觀察事實、已嘗試方法與結果、採用修法、驗證結果、可重用 heuristic 與產品改進建議的 issue 草稿。
2. **Given** 該案例屬於一次性本機雜訊或已知重複問題, **When** 使用者評估是否建立 issue, **Then** 系統應支援不建立新 issue，並清楚保留不開單的原因。
3. **Given** 系統產生 issue 草稿, **When** 草稿中包含使用者名稱、絕對路徑、代理/帳號資訊或疑似敏感字串, **Then** 草稿內容應預設以遮罩後形式呈現，並允許人類在發布前進一步刪除或調整敏感細節。

### Edge Cases

- 當診斷顯示工具鏈正常，但實際問題來自裝置、線材、權限或連線時，系統必須清楚提醒這次修復流程只涵蓋工具鏈與環境 readiness。
- 當同一組證據可對應多個合理問題類型時，系統必須突出主要推薦流程，但仍需保留次要可疑因素，避免過度武斷。
- 當某個修復步驟只能部分在介面內完成時，系統必須清楚區分「可直接執行的步驟」與「仍需使用者手動完成的步驟」，避免讓使用者誤以為整條流程已全自動處理。
- 當使用者在輕量確認階段取消 `自動修復` 時，系統必須保留已診斷出的推薦流程與證據，但不得留下模糊的「半開始」狀態。
- 當某個互動式修復操作失敗、中斷或只完成部分工作時，系統必須保留可理解的結果狀態，並提供下一步，而不是回到模糊的初始狀態。
- 當單次 `自動修復` 已經在主要流程中遇到阻塞失敗時，系統不得無說明地繼續嘗試其他不屬於該流程的修復路徑。
- 當同一個修復步驟被重試多次時，系統必須能讓使用者分辨最新一次結果與先前歷程，避免混淆哪些方法其實已經失敗過。
- 當工作區環境指紋已明顯改變（例如核心工具位置或重要設定已變更）時，系統必須避免把舊歷程誤當成仍然完全適用，並應提示使用者重新確認或重置歷程。
- 當所有推薦修復路徑都失敗時，系統必須仍能輸出一份清楚的「已試過哪些方法、各自結果、仍未嘗試哪些方向」摘要，讓後續 AI 或維護者接手時不需重頭試錯。
- 當 AI 想提出摘要中沒有證據支持的修法時，摘要必須引導 AI 明說缺少哪些資訊，而不是憑空補完。
- 當同一個案例的修復內容包含使用者名稱、工作區路徑或代理資訊時，issue 草稿流程必須允許人類先完成隱私審查再公開。
- 當自動遮罩可能影響後續排查時，系統必須讓使用者清楚知道哪些資訊已被遮罩，並可在明確操作下選擇揭露更多細節，而不是默默混用遮罩版與原文版。
- 當候選 issue 與既有問題高度重疊時，系統應優先鼓勵引用或補充既有追蹤，而不是再新增一張幾乎相同的 issue。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 在既有的 PlatformIO 狀態檢查體驗中，同時提供診斷結果與修復引導，而不是要求使用者切換到另一個獨立工具才能知道下一步。
- **FR-002**: System MUST 將異常診斷結果分類為少量、可理解、可行動的問題類型，讓使用者能知道目前最可能是哪一類環境問題。
- **FR-003**: System MUST 對每一個需要處理的主要問題類型提供推薦修復流程，且至少包含主要修法、備援修法與修後驗證方式。
- **FR-004**: System MUST 在安全、可說明、可控制的範圍內，提供可由單一主 `自動修復` 按鈕啟動的修復操作，讓使用者能直接讓系統先嘗試最推薦的部分修復步驟；第一版僅限於使用者空間內可完成的修復。
- **FR-005**: System MUST 在使用者執行 `自動修復` 前，先提供一次輕量確認，清楚說明將要做的事情、有限步驟序列、影響範圍與執行邊界，避免靜默改動環境。
- **FR-006**: System MUST 對每一次互動式修復操作提供進度、結果與後續建議，讓使用者知道哪些步驟已完成、哪些仍需手動處理。
- **FR-007**: System MUST 將單一主 `自動修復` 按鈕實際涵蓋的步驟與仍需外部手動完成的步驟明確區分，避免把半自動流程偽裝成全自動修復。
- **FR-008**: System MUST 在顯示推薦修復流程的同時保留底層診斷證據，讓使用者與協助者可以理解系統為何推薦這條路徑。
- **FR-009**: System MUST 允許使用者在嘗試修復後重新執行檢查，並清楚指出目前狀態是改善、未變或惡化。
- **FR-010**: System MUST 將每一次互動式修復嘗試記錄成可讀的修復歷程，至少包含嘗試了哪一個方法、結果是成功/失敗/中止/未定，以及接下來建議的下一步。
- **FR-011**: System MUST 在推薦下一條修復路徑時參考先前已嘗試過的方法，避免無說明地重複要求使用者再次嘗試同一條已知失敗路徑。
- **FR-012**: System MUST 以工作區為範圍持久保存修復歷程，使其可跨面板重開與 VS Code 重啟保留，直到使用者主動清除或系統判定環境指紋已明顯改變。
- **FR-013**: System MUST 提供可複製的結構化修復摘要，讓使用者可把結果直接貼給 AI、教師、維護者或其他協助者。
- **FR-014**: System MUST 在修復摘要中明確區分不可變事實、已嘗試修復歷程與要求 AI 執行的分析任務，避免協助者忽略證據邊界。
- **FR-015**: System MUST 要求 AI 回覆固定欄位，至少包括已觀察事實、最可能根因、實際採用修法、為何選這條修法、驗證方式、可重用 heuristic、接下來仍可嘗試的方法與產品改進建議。
- **FR-016**: System MUST 讓修復摘要在不同語系下仍保有穩定欄位結構，以支援後續知識整理、比較與 triage。
- **FR-017**: System MUST 在不依賴 AI 的情況下仍保有足夠的人類可讀性，讓教師、維護者或協作者能直接理解摘要內容。
- **FR-018**: System MUST 在 AI 修復封包與 issue 草稿中預設自動遮罩已知敏感資訊，例如使用者名稱、絕對路徑、代理/帳號資訊與疑似 token；若需揭露更多細節，必須由使用者明確選擇。
- **FR-019**: System MUST 支援在成功修復或識別到穩定 heuristic 後建立後續追蹤草稿，但只限於被判定值得產品化沉澱的案例。
- **FR-020**: System MUST 不得在互動式修復流程結束後，未經明確人類核准就自動建立公開 issue。
- **FR-021**: System MUST 在 issue 建立前要求進行最基本的重複檢查與隱私檢查，避免公開 backlog 被一次性雜訊或敏感資訊污染。
- **FR-022**: System MUST 讓 issue 草稿清楚標示哪些內容來自 AI 協助整理、哪些內容已經過人類確認，並納入已嘗試方法與其結果。
- **FR-023**: System MUST 區分「值得沉澱成產品/UX 改進」的案例與「僅屬本機一次性環境雜訊」的案例，避免不必要的公開追蹤。
- **FR-024**: System MUST 在診斷結果為完全正常時避免強迫使用者進入修復流程，而是清楚說明目前沒有需要處理的工具鏈問題。
- **FR-025**: System MUST 在推薦修復流程中明確提醒適用範圍與邊界，避免使用者誤以為所有上傳、裝置或連線問題都能透過本流程解決。
- **FR-026**: System MUST 不得透過 `自動修復` 按鈕靜默執行高風險、不可逆、超出工具鏈 readiness 範圍、或需要系統層權限的變更。
- **FR-027**: System MUST 在某個自動修復步驟不適用於目前平台、權限或環境時，明確告知不可用原因並提供替代手動路徑。
- **FR-028**: System MUST 不得在第一版透過 `自動修復` 按鈕執行系統套件管理器、修改 shell profile、安裝系統 Python，或要求系統層級永久性設定變更。
- **FR-029**: System MUST 將單次 `自動修復` 執行限制在同一條主要修復流程中的有限步驟序列，並在達成成功條件或遇到阻塞失敗時停止，而不是自動掃過所有可能修復路徑。
- **FR-030**: System MUST 將使用者確認限制為每次 `自動修復` 執行前的一次輕量確認，而不是要求每個步驟逐一確認；若使用者未確認，系統不得開始執行任何修復步驟。

### Key Entities *(include if feature involves data)*

- **Diagnostic Finding**: 代表單一診斷觀察結果的資訊單位，包含狀態、證據、說明、可信度與相關可疑因素。
- **Repair Flow**: 代表系統針對某一類問題推薦的修復路徑，包含主要修法、備援修法、驗證方式與適用範圍提醒。
- **Auto Repair Run**: 代表由主 `自動修復` 按鈕啟動的一次修復執行，包含同一條主要修復流程中的有限步驟序列、執行前確認內容、執行條件、執行結果與下一步。
- **Repair Attempt Record**: 代表單次修復嘗試的歷程紀錄，包含方法、執行結果、可疑原因、是否值得重試、對下一步建議的影響，以及與工作區環境指紋對應的保存範圍。
- **AI Repair Packet**: 代表可分享給 AI 或協助者的結構化摘要，包含證據區塊與回答契約區塊。
- **Issue Draft Proposal**: 代表在人類審查前準備好的追蹤草稿，包含問題摘要、修法、驗證結果、可重用 heuristic、產品改進建議、審查註記與遮罩後的敏感資訊處理結果。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在代表性 degraded 或 unavailable 情境中，90% 以上的使用者可於 2 分鐘內從既有 PlatformIO 狀態檢查找出一個明確的下一步，且在適用情況下可直接點擊主 `自動修復` 按鈕啟動至少一個安全修復嘗試，而不需要自行解讀原始日誌。
- **SC-002**: 90% 以上的異常診斷結果會在同一個檢查介面中同時提供主要問題分類、推薦修復流程、主 `自動修復` 動作（若適用）與驗證方式。
- **SC-003**: 使用結構化修復摘要求助時，85% 以上的 AI 或遠端協助回覆能在第一次回應中包含根因判斷、採用修法、驗證方式與可重用 heuristic。
- **SC-004**: 與 PlatformIO 環境判讀相關的支援往返次數，在釋出後的樣本觀察中可下降 50% 以上。
- **SC-005**: 對於經人類核准而建立的後續 issue 草稿，90% 以上能在不需大幅重寫的情況下進入維護者 triage 流程。
- **SC-006**: 在被評估為 issue 候選的案例中，90% 以上能成功避免把一次性雜訊、重複案例或未審查的敏感資訊直接送入公開 backlog。
- **SC-007**: 在可自動修復的代表性案例中，80% 以上的使用者能在同一個檢查介面內透過主 `自動修復` 按鈕完成至少一個安全修復步驟並成功理解接下來仍需手動處理的部分。
- **SC-008**: 在需要轉交 AI 或維護者接手的失敗案例中，90% 以上的修復摘要會明確列出已嘗試方法、各自結果與仍可繼續嘗試的方向，讓接手者不需從頭重複試錯。

## Assumptions

- 使用者會從既有的 PlatformIO 狀態檢查入口進入本功能，而不是期待另一套新的診斷入口。
- 第一版聚焦於分流式修復引導、由單一主 `自動修復` 按鈕啟動的安全自動修復嘗試、AI-ready 摘要與人工核准的 issue 草稿，不包含全自動環境修復或無人值守的公開開單。
- 即使使用者會向 AI 求助，最終是否將修復經驗沉澱成公開 issue 仍需有人類把關。
- 維護者更需要高訊號、可重現、可產品化的案例，而不是紀錄每一台機器的一次性環境雜訊。
- 使用者與維護者願意在公開追蹤前檢查並移除不適合公開的本機資訊。
- AI 修復封包與 issue 草稿在預設情況下會先遮罩已知敏感資訊；若排查需要更多細節，將由使用者主動決定是否揭露。
- 本功能主要處理工具鏈 readiness 與診斷/修復溝通品質，不保證能直接排除所有裝置、權限、線材或連線層級問題。
- `自動修復` 按鈕只涵蓋低風險、可說明、可控制、且能在使用者空間內完成的步驟；超出此範圍的修復仍以引導式手動流程為主。
- 修復歷程會以工作區為範圍持久保存，但當環境指紋明顯改變時，系統可要求使用者確認是否沿用或重置既有歷程。
- 修復歷程的核心目的是提升後續判斷與溝通品質；即使最終仍需 AI 或維護者介入，也應能利用這份歷程避免重複試錯並產出更好的 issue 報告。
