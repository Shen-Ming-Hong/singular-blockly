# Feature Specification: 鎖定函式積木 (Lock Function Block)

**Feature Branch**: `050-lock-function-block`
**Created**: 2026-04-08
**Status**: Draft
**Input**: 在對著已建立函式積木點右鍵選單中加入「鎖定積木」選項，鎖定後函式內容無法更動但仍可呼叫，視覺上有外觀變化，支援 Arduino 自訂函式與 MicroPython 內建函式，向下相容舊版擴充套件。

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 老師鎖定函式防止學生誤刪 (Priority: P1)

老師預先建立好函式積木（例如「驅動馬達向前」），並在課堂前將函式鎖定。學生打開專案後可以正常拖曳該函式的呼叫積木使用，但無法修改函式內的任何程式邏輯、重新命名、新增或移除參數，也無法刪除整個函式定義積木。

**Why this priority**: 這是此功能的核心使用情境，直接解決學生課堂中誤刪或誤改教師預設邏輯的問題。

**Independent Test**: 建立一個函式積木 → 右鍵鎖定 → 嘗試四種修改操作（重新命名、增刪參數、STACK 增刪積木、刪除積木），全部應被阻止 → 函式呼叫積木仍可正常拖入使用並產生程式碼。

**Acceptance Scenarios**:

1. **Given** 工作區中存在一個 `arduino_function` 積木，**When** 使用者對其按右鍵並選擇「鎖定積木」，**Then** 積木外觀改變（顏色變化 + 🔒 圖示），且四種修改操作均被阻止。
2. **Given** 函式積木已鎖定，**When** 使用者嘗試在 STACK 區域拖入新積木，**Then** 拖入操作被自動拒絕，新積木回到原位。
3. **Given** 函式積木已鎖定，**When** 使用者嘗試點擊函式名稱欄位修改名稱，**Then** 欄位不可編輯。
4. **Given** 函式積木已鎖定，**When** 使用者嘗試打開 mutator（齒輪圖示）修改參數，**Then** 修改不會套用至積木。
5. **Given** 函式積木已鎖定，**When** 使用者嘗試刪除該積木（鍵盤 Delete 或右鍵刪除），**Then** 刪除操作被阻止。
6. **Given** 函式積木已鎖定，**When** 使用者將函式呼叫積木拖入 `arduino_setup_loop`，**Then** 程式碼正常生成。

---

### User Story 2 - 鎖定狀態持久化（存檔重開） (Priority: P2)

老師鎖定函式後儲存專案，學生下次開啟同一個 `.json` 專案檔案時，函式積木應自動恢復為鎖定狀態，不需要重新手動鎖定。

**Why this priority**: 若鎖定狀態不能持久化，老師每次開課前都需要重新鎖定，大幅降低功能實用性。

**Independent Test**: 鎖定函式 → 存檔（Ctrl+S）→ 關閉再重開專案 → 確認積木顯示為鎖定狀態且所有限制仍有效。

**Acceptance Scenarios**:

1. **Given** 函式積木已鎖定，**When** 使用者儲存專案後重新開啟，**Then** 函式積木自動恢復鎖定狀態（外觀與操作限制均恢復）。
2. **Given** 未鎖定的函式積木，**When** 儲存並重開，**Then** 積木保持未鎖定狀態（無副作用）。

---

### User Story 3 - 解鎖函式恢復編輯 (Priority: P2)

老師或使用者需要修改已鎖定的函式時，可以對積木右鍵選擇「解鎖積木」，恢復正常的可編輯狀態。

**Why this priority**: 鎖定功能的設計目的是防誤操作非永久封鎖，老師需要保留修改函式的能力。

**Independent Test**: 鎖定函式 → 右鍵解鎖 → 嘗試重新命名 → 操作應成功。

**Acceptance Scenarios**:

1. **Given** 函式積木已鎖定，**When** 使用者右鍵選擇「解鎖積木」，**Then** 積木外觀恢復正常（顏色與 🔒 圖示消失），所有編輯操作重新可用。
2. **Given** 已解鎖的函式積木，**When** 右鍵選單中，**Then** 顯示「鎖定積木」而非「解鎖積木」（toggle 行為）。

---

### User Story 4 - CyberBrick（MicroPython）函式鎖定 (Priority: P3)

使用 CyberBrick 板（MicroPython 模式）時，內建的函式定義積木同樣支援鎖定功能，行為與 Arduino 函式一致。

**Why this priority**: 擴充至 MicroPython 拓展使用情境，但核心功能已由 P1–P2 覆蓋，不影響 MVP。

**Independent Test**: 切換至 CyberBrick 板 → 建立 MicroPython 函式 → 鎖定 → 驗證限制與視覺效果。

**Acceptance Scenarios**:

1. **Given** CyberBrick 板模式下存在 `procedures_defnoreturn` 或 `procedures_defreturn` 積木，**When** 使用者右鍵選擇「鎖定積木」，**Then** 積木鎖定且行為與 Arduino 函式一致。
2. **Given** MicroPython 函式已鎖定，**When** 存檔並重開，**Then** 鎖定狀態恢復。

---

### User Story 5 - 舊版擴充套件向下相容 (Priority: P1)

尚未更新擴充套件的使用者，打開包含鎖定函式積木的專案（例如老師分享的範本）時，工作區應正常載入，函式積木以未鎖定狀態顯示並可正常編輯，不發生任何錯誤或崩潰。

**Why this priority**: 若舊版無法載入含鎖定積木的檔案，會造成師生協作斷裂，影響所有使用新功能的早期用戶。

**Independent Test**: 用新版建立含鎖定函式的專案並存檔 → 在不含新鎖定邏輯的舊版環境下開啟該 JSON → 確認工作區正常載入，積木可正常使用。

**Acceptance Scenarios**:

1. **Given** 舊版擴充套件（不含鎖定功能），**When** 開啟含鎖定 `arduino_function` 的 JSON 檔，**Then** 工作區正常載入，函式積木以未鎖定狀態顯示，無錯誤訊息。
2. **Given** 舊版擴充套件，**When** 開啟含鎖定 `procedures_defnoreturn` 的 JSON 檔，**Then** 工作區正常載入，函式積木以未鎖定狀態顯示，無錯誤訊息。
3. **Given** 舊版擴充套件正常載入含鎖定函式的專案，**When** 使用者以舊版編輯後存檔，**Then** 鎖定資訊被清除（因舊版不寫入），此為預期行為。

---

### Edge Cases

- 函式積木已鎖定時，使用者透過選單「全選」後刪除，鎖定的函式定義積木應不被刪除，其餘可刪除積木正常刪除。
- 工作區有多個函式積木時，鎖定操作各自獨立，不互相影響。
- 函式積木鎖定時，STACK 內已有的積木維持現有狀態（不被移除），但不可再被修改或刪除。
- 切換板子（Arduino/CyberBrick）導致工作區清空時，鎖定狀態隨積木消失，不留殘留。
- 複製貼上一個鎖定的函式積木時，貼上後的副本保持鎖定狀態。

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系統 MUST 在 `arduino_function` 積木的右鍵選單中提供「鎖定積木」選項，且只在該積木類型上顯示。
- **FR-002**: 系統 MUST 在 `procedures_defnoreturn` 及 `procedures_defreturn` 積木的右鍵選單中同樣提供「鎖定積木」選項。
- **FR-003**: 鎖定後，使用者 MUST NOT 能重新命名函式（函式名稱欄位不可編輯）。
- **FR-004**: 鎖定後，使用者 MUST NOT 能透過 mutator UI 新增或移除函式參數（變更不套用）。
- **FR-005**: 鎖定後，使用者 MUST NOT 能在函式 STACK 區域新增積木（拖入被自動拒絕）。
- **FR-006**: 鎖定後，使用者 MUST NOT 能刪除 STACK 區域內的現有積木。
- **FR-007**: 鎖定後，使用者 MUST NOT 能刪除整個函式定義積木。
- **FR-008**: 鎖定後，函式呼叫積木 MUST 仍可正常使用、移動、刪除並生成正確的程式碼（鎖定不限制呼叫積木的任何操作）。
- **FR-009**: 鎖定狀態 MUST 序列化至工作區 JSON 檔案，重開檔案後自動恢復。
- **FR-010**: 已鎖定的積木右鍵選單 MUST 顯示「解鎖積木」，選擇後恢復所有編輯能力（toggle 行為）。
- **FR-011**: 鎖定的函式積木 MUST 以視覺方式區別於未鎖定積木：顏色改變（灰褐色）且顯示 🔒 圖示。
- **FR-012**: 功能文字（「鎖定積木」、「解鎖積木」）MUST 支援全部 15 個語系（i18n）。
- **FR-013**: 舊版擴充套件（不含此功能）MUST 能正常載入含鎖定函式積木的 JSON 檔，函式積木以未鎖定狀態顯示，不崩潰。
- **FR-014**: 鎖定與解鎖操作 MUST NOT 進入 Blockly undo 佇列，不可被 Ctrl+Z 復原（防止學生透過復原繞過鎖定）。
- **FR-015**: 複製貼上一個已鎖定的函式積木時，貼上的副本 MUST 保持鎖定狀態（防止學生透過複製副本再修改以繞過保護）。
- **FR-016**: 鎖定後，STACK 內子積木的非修改性右鍵操作（說明、折疊/展開等）MUST 仍可使用；僅結構性修改操作（刪除、移動、複製後貼回 STACK 等）受限制。

### Key Entities

- **鎖定狀態（Lock State）**: 附加於函式定義積木的布林值屬性，代表積木是否處於鎖定模式。序列化至存檔 JSON，反序列化時恢復並套用對應的外觀與行為限制。
- **函式定義積木（Function Definition Block）**: 工作區中可被鎖定的目標積木，包含 `arduino_function`（Arduino 板）、`procedures_defnoreturn`、`procedures_defreturn`（MicroPython 板）三種類型。
- **函式呼叫積木（Function Call Block）**: 對應函式定義積木的呼叫端，鎖定時不受影響，可正常建立與使用。

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 教師可在 30 秒內完成一個函式積木的鎖定操作（右鍵 → 選擇鎖定）。
- **SC-002**: 鎖定後，100% 的修改操作（重新命名、增刪參數、STACK 增刪積木、刪除積木）均被阻止，無任何可繞過路徑。
- **SC-003**: 鎖定狀態序列化正確率 100%——儲存後重開的專案，所有原本鎖定的函式均恢復鎖定狀態。
- **SC-004**: 舊版擴充套件載入含鎖定積木的專案，成功率 100%（不崩潰、不顯示錯誤）。
- **SC-005**: 功能文字於全部 15 個語系均有翻譯，`npm run validate:i18n` 通過率 100%。
- **SC-006**: 鎖定/解鎖切換後，積木外觀在 500ms 內完成更新（顏色 + 圖示），無需使用者手動刷新。

## Assumptions

- 鎖定功能不需要密碼保護，任何人皆可透過右鍵解鎖（設計目標為防止學生誤操作，非安全防護）。
- 鎖定積木內已存在的 STACK 積木在鎖定後維持現有狀態，不自動移除。
- 函式呼叫積木的外觀不需要額外的鎖頭圖示標記（僅定義積木有視覺變化）。

## Out of Scope

- 密碼保護或權限管理機制
- 函式呼叫積木的視覺鎖定標記
- 部分鎖定（例如只鎖定名稱但允許修改 STACK）
- 鎖定變數積木或其他非函式類型積木

## Clarifications

### Session 2026-04-08

- Q: 鎖定/解鎖操作是否應可被 Ctrl+Z 復原？ → A: 不可復原（直接狀態變更，不進 undo 佇列）
- Q: 複製貼上已鎖定的函式積木，副本應為鎖定還是解鎖狀態？ → A: 副本保持鎖定狀態
- Q: 鎖定後 STACK 內子積木的非修改性右鍵操作（說明、折疊等）是否應被允許？ → A: 允許（僅阻止結構性修改，不限制所有互動）
- Q: 工具箱 flyout 中的函式呼叫積木是否需要顯示鎖定提示？ → A: 不顯示（保持現有 flyout 外觀，Out of Scope 範圍確認）
- Q: 函式定義積木鎖定後，工作區中已存在的呼叫積木是否可被刪除？ → A: 可以被刪除（鎖定只保護定義積木邏輯，不限制學生的程式流程設計）
