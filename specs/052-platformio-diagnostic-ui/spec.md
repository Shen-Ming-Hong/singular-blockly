# 功能規格：PlatformIO 診斷 UI

**功能分支**：`[052-platformio-diagnostic-ui]`  
**建立日期**：2026-05-12  
**最後更新**：2026-05-13  
**狀態**：待實作  
**輸入**：使用者需求：「將舊版 command/progress 診斷方案改為獨立診斷頁／診斷面板，保持獨立 command 入口，顯示完整清單與清楚原因／下一步，第一版只保留重新測試與複製診斷摘要，並保護既有 path fallback 與 upload button icon 修正基線。」

## 澄清紀錄

### 2026-05-12 澄清

- Q: 第一版 PlatformIO 診斷 UI 必須固定顯示哪些項目？ → A: 顯示 `pio`、`penv` 根目錄、`python`、`pip`、`mpremote`。
- Q: `mpremote` 是否也應視為透過 `penv` 提供的工具？ → A: 是，第一版診斷應將 `mpremote` 與 `python`、`pip` 一樣視為 PlatformIO `penv` 工具鏈的一部分，並清楚呈現其來源。

### 2026-05-13 澄清

- Q: 第一版診斷資訊要放在哪種 UI 載體？ → A: 使用獨立的診斷頁／診斷面板，讓使用者能停留、回看與重新測試；不是 Quick Pick，也不是只靠通知摘要。
- Q: 入口要不要綁進 upload 流程？ → A: 不要。維持獨立 command 入口，讓使用者可以在上傳前主動開啟診斷。
- Q: 第一版面板內保留哪些操作？ → A: 只保留 `重新測試` 與 `複製診斷摘要`，避免過早擴張成設定或修復中心。
- Q: 第一版是否提供手動設定 `pio` / `penv` / `python` / `pip` / `mpremote` 路徑？ → A: 不提供，這屬於下一次 SDD 候選。
- Q: 視覺風格是否可以另起一套？ → A: 不可以。需延續本專案既有 WebView / panel / modal 的設計語言，並在 light / dark theme 下都清楚易讀。

## 使用者情境與測試（必要）

### 使用者故事 1 - 在獨立診斷頁看到完整工具清單（優先級：P1）

作為需要上傳或監看裝置的使用者，我想從一個獨立的診斷頁／診斷面板查看 `pio`、`penv` 根目錄、`python`、`pip`、`mpremote` 的完整狀態清單，這樣我可以在真正操作前先理解本機工具鏈是否可用，而不是等到流程失敗後才猜原因。

**優先順序原因**：這是整個功能的核心價值。只要使用者能先看見完整清單與整體狀態，就能大幅降低「明明安裝了卻不知道系統實際採用哪個位置」的困惑。

**獨立驗證方式**：執行獨立診斷 command 後，系統應開啟獨立診斷頁／診斷面板，並一次顯示五個固定項目與整體狀態；不需要進入 upload 流程，也不應只看到短暫通知摘要。

**驗收情境**：

1. **前提** 使用者已開啟專案，**當** 使用者執行 PlatformIO 診斷 command，**則** 系統應開啟獨立診斷頁／診斷面板並顯示五個固定項目的完整清單。
2. **前提** 某些工具已找到、某些工具未找到，**當** 診斷結果顯示完成，**則** 每個工具都應有清楚的可用性標示、實際位置與來源說明。
3. **前提** 使用者再次執行診斷 command，**當** 診斷頁／診斷面板已存在，**則** 系統應重新帶回同一個診斷介面，而不是分散成多個互相競爭的入口。
4. **前提** 使用者正在查看診斷結果，**當** 使用者瀏覽內容，**則** 介面應保留完整清單，不應要求使用者透過 Quick Pick 或僅靠通知文字逐步拼湊資訊。

---

### 使用者故事 2 - 理解錯誤原因並重新測試（優先級：P2）

作為遇到環境問題的使用者，我想在同一個診斷頁／診斷面板裡看到清楚的錯誤原因與下一步建議，並能直接重新測試，這樣我可以修正環境後立即確認問題是否真的排除。

**優先順序原因**：只知道「失敗」還不夠；使用者需要知道為什麼失敗，以及下一步應該做什麼。重新測試則能把「看懂問題」與「確認修正」串成一個完整閉環。

**獨立驗證方式**：在存在失敗或警告項目的情境中，介面應為每個需要處理的項目顯示可讀原因與下一步建議；按下 `重新測試` 後，結果應更新為最新狀態，而不是沿用舊資訊。

**驗收情境**：

1. **前提** 某個工具不存在、無法執行或來源異常，**當** 使用者查看診斷結果，**則** 系統應提供具體且可理解的失敗原因。
2. **前提** 某個工具需要使用者處理，**當** 使用者查看該項目，**則** 系統應顯示清楚的下一步建議，而不是只留下抽象的錯誤碼。
3. **前提** 使用者已調整本機環境，**當** 使用者按下 `重新測試`，**則** 系統應重新檢查五個固定項目並更新結果。
4. **前提** 診斷過程中只有部分項目失敗，**當** 結果顯示完成，**則** 系統仍應保留整份完整清單，而不是因單一失敗中止其他資訊呈現。

---

### 使用者故事 3 - 複製並分享診斷摘要（優先級：P3）

作為需要向教師、維護者或協作者求助的使用者，我想從診斷頁／診斷面板直接複製可讀的診斷摘要，這樣我可以快速提供一致且足夠的資訊，減少來回追問。

**優先順序原因**：這能把診斷結果從「只存在畫面上」變成可分享的支援資訊，直接降低遠端協助與 issue triage 的成本。

**獨立驗證方式**：診斷結果產生後，按下 `複製診斷摘要` 應能得到固定格式且易讀的純文字內容，其中包含整體狀態、五個固定項目與關鍵原因摘要。

**驗收情境**：

1. **前提** 診斷結果已顯示於診斷頁／診斷面板，**當** 使用者按下 `複製診斷摘要`，**則** 系統應提供包含工具狀態、位置、原因摘要與範圍提醒的可分享內容。
2. **前提** 診斷中存在失敗或警告項目，**當** 使用者分享摘要，**則** 摘要應指出主要問題集中在哪些工具以及建議的下一步方向。
3. **前提** 使用者重新測試過，**當** 使用者再次複製摘要，**則** 摘要應反映最新結果，而不是過期內容。

### 邊界情境

- 當所有工具都顯示可用，但使用者仍有上傳問題時，系統應明確提醒本次診斷僅涵蓋工具辨識與可用性，不代表裝置、連線、線材或權限一定正常。
- 當工具檔案存在但無法正常執行時，系統應將其視為需要處理的項目，而不是單純標示為成功。
- 當使用者沒有權限讀取某個工具位置時，系統應顯示可理解的原因與下一步建議，而不是空白狀態。
- 當結果來源改變（例如同名工具來自不同位置）時，系統應清楚呈現目前實際採用的位置與來源，避免使用者誤以為系統仍沿用舊設定。
- 當使用者反覆開啟、關閉、重開診斷頁／診斷面板時，系統不應把資訊拆成多個互相衝突的 UI 入口。
- 當使用者開啟或關閉診斷頁／診斷面板、重新測試，或返回 Blockly 編輯器時，既有 upload button icon 狀態不得被非預期重設，也不得重新出現自動旋轉的回歸問題。

## 需求（必要）

### 功能需求

- **FR-001**: 系統 MUST 提供一個獨立 command 入口，讓使用者可在不開始上傳流程的情況下開啟 PlatformIO 診斷。
- **FR-002**: 系統 MUST 以獨立診斷頁／診斷面板呈現 PlatformIO 診斷結果，並在重複開啟時回到同一個診斷介面。
- **FR-003**: 系統 MUST 以完整清單顯示第一版固定診斷項目 `pio`、`penv` 根目錄、`python`、`pip`、`mpremote`，不可只依賴 Quick Pick 或通知摘要作為主要資訊載體。
- **FR-004**: 系統 MUST 為每個工具顯示目前狀態、實際使用的位置資訊與結果來源說明。
- **FR-005**: 系統 MUST 清楚區分成功、警告與失敗狀態，讓使用者一眼看懂哪些項目需要處理。
- **FR-006**: 系統 MUST 在項目失敗或需要注意時顯示具體且可讀的原因說明，避免只顯示模糊的「找不到」或單一錯誤碼。
- **FR-007**: 系統 MUST 在項目為警告或失敗時提供清楚的下一步建議，幫助使用者知道下一步該做什麼。
- **FR-008**: 系統 MUST 在第一版中只提供 `重新測試` 與 `複製診斷摘要` 兩個主要操作，保持操作簡潔。
- **FR-009**: 系統 MUST 在使用者按下 `重新測試` 後重新執行檢查並更新顯示結果，不可沿用舊結果冒充最新狀態。
- **FR-010**: 系統 MUST 讓使用者能從診斷頁／診斷面板取得可分享的診斷摘要，以支援回報問題或尋求協助。
- **FR-011**: 系統 MUST 在部分工具成功、部分工具失敗的情況下仍保留完整結果，而不是因單一失敗而中止全部資訊呈現。
- **FR-012**: 系統 MUST 以與現有專案 WebView / panel / modal 一致的視覺語言呈現診斷內容，讓卡片、區塊、狀態色與操作按鈕在 light / dark theme 下都保持清楚且一致。
- **FR-013**: 系統 MUST 明確提醒診斷範圍僅涵蓋工具辨識與可用性，不將裝置連線或其他外部因素誤判為工具問題。
- **FR-014**: 系統 MUST 保留目前已落地的 executableResolver / uploader path fallback 修正與 upload button icon regression 修正作為既有基線，不因新增診斷 UI 而改變既有路徑判定結果，也不得重新引入 upload icon 非預期旋轉的回歸。
- **FR-015**: 系統 MUST 在診斷 `python`、`pip`、`mpremote` 時，明確表達它們是否來自偵測到的 PlatformIO `penv` 工具鏈。
- **FR-016**: 系統 MUST 在使用者執行診斷 command 後，於約 1 秒內顯示可見的 loading 狀態，並在 10 秒內完成診斷結果顯示，或以可理解的 timeout / degraded 狀態離開 loading，而不是讓使用者無限等待。

### 關鍵實體

- **Diagnostic Session**: 一次由使用者開啟或重新測試所產生的完整診斷結果集合，包含檢查時間、整體狀態與各工具結果。
- **Tool Check Result**: 單一工具的診斷項目，包含工具名稱、可用性狀態、實際位置、來源說明、是否屬於偵測到的 `penv` 工具鏈、失敗原因與下一步建議。
- **Diagnostic Panel View**: 使用者在獨立診斷頁／診斷面板看到的完整呈現，包含整體摘要、工具清單、範圍提醒與兩個主要操作。
- **Diagnostic Summary**: 提供給使用者分享或回報問題的精簡結果，包含必要工具狀態、關鍵原因與範圍提醒。

## 成功標準（必要）

### 可衡量成果

- **SC-001**: 90% 以上的使用者可在 1 分鐘內透過獨立診斷入口開啟診斷頁／診斷面板，並判斷目前工具鏈是否大致可用。
- **SC-002**: 95% 以上的診斷執行可對五個固定項目產生明確狀態，且每個警告或失敗項目都附帶可理解的原因與下一步建議。
- **SC-003**: 在測試情境中，90% 以上的使用者能只依靠診斷頁／診斷面板內容指出「找不到 PlatformIO」或工具鏈不完整的主要原因，不需閱讀原始日誌或程式碼。
- **SC-004**: 在手動驗證情境中，90% 以上的使用者能在同一個診斷頁／診斷面板內完成重新測試與複製摘要兩項主要操作，不需切換到其他設定或流程。
- **SC-005**: 與 PlatformIO 或相關工具偵測有關的支援往返次數，在功能釋出後可降低 50% 以上。
- **SC-006**: 在代表性驗證情境中，95% 以上的診斷執行可於約 1 秒內顯示 loading UI，並在 10 秒內完成結果顯示或以 timeout / degraded 狀態結束 loading。

## 假設

- 這個功能面向的是需要使用上傳、監看或相關工具流程的使用者，而不是純閱讀積木的使用者。
- 第一版聚焦於**獨立診斷頁／診斷面板**中的工具辨識、來源說明、原因／下一步提示與主動重新測試，不提供修復精靈或進階設定。
- 使用者的環境可能同時存在多個可疑似匹配的位置，系統需要清楚呈現最終採用的判定結果與來源。
- 對 CyberBrick 相關流程而言，`python`、`pip`、`mpremote` 預設被視為由 PlatformIO `penv` 提供的工具鏈成員；若來源不同，診斷結果仍需明確標示。
- 目前已落地的路徑 fallback 與 upload button 修正被視為既有基線；本功能新增的是可見性與診斷 UX，而不是重新定義整個上傳流程。
- 手動設定 `pio` / `penv` / `python` / `pip` / `mpremote` 路徑屬於下一次 SDD 的候選範圍，不納入本次 v1。
